# 用户管理系统需求分析文档

**项目名称**：用户管理系统
**文档版本**：v1.0
**创建日期**：2026-03-04
**技术栈**：Go + MySQL + HTML5

---

## 1. 项目概述

### 1.1 项目背景

本项目是一个基于 Web 的用户管理系统，旨在提供用户注册、登录、权限管理等功能，支持管理员和普通用户两种角色，满足企业内部或团队协作的用户管理需求。

### 1.2 项目目标

- 提供完整的用户生命周期管理（注册、登录、信息修改、删除）
- 实现基于角色的权限控制（RBAC）
- 保证用户数据安全（密码加密存储、认证授权）
- 提供良好的用户体验（前后端分离、响应式界面）

## 2. 功能需求

### 2.1 用户注册

**功能描述**：
用户可以通过前端表单注册新账户，系统自动验证用户信息的合法性，并将用户数据存入数据库。

**功能点**：
- 用户名注册
  - 长度限制：6-13个字符
  - 格式要求：仅允许字母和数字组合
  - 唯一性检查：用户名不能重复

- 密码设置
  - 长度限制：6-13个字符
  - 格式要求：仅允许字母和数字组合
  - 密码加密存储（防止明文泄露）

- 邮箱设置（可选）
  - 邮箱格式校验

**流程图**：
```
用户提交注册信息
  ↓
前端参数校验
  ↓
后端参数校验
  ↓
用户名唯一性检查
  ↓
密码加密
  ↓
存入数据库
  ↓
返回注册成功/失败
```

### 2.2 用户登录

**功能描述**：
用户通过用户名和密码登录系统，系统验证用户身份并返回访问凭证（Token）。

**功能点**：
- 用户名密码校验
  - 用户名存在性检查
  - 密码匹配验证
- 身份认证
  - 生成 JWT Token
  - 返回用户信息（含角色）
- 登录状态管理
  - 前端存储 Token
  - Token 自动刷新机制

**返回信息**：
```json
{
  "success": true,
  "data": {
    "id": "1",
    "username": "admin",
    "role": "admin",
    "token": "eyJhbGciOiJIUzI1NiIs..."
  }
}
```

### 2.3 用户管理（管理员）

**功能描述**：
管理员可以管理系统中所有用户，包括查看、编辑、删除用户信息。

**权限范围**：
- 查看所有用户列表
- 编辑任意用户的个人信息（用户名、邮箱、头像）
- 删除任意用户账户
- 修改用户角色（授予/收回管理员权限）
- 修改用户状态（启用/禁用）

**操作流程**：
```
管理员登录系统
  ↓
访问用户管理页面
  ↓
查看用户列表（显示用户名、角色、状态）
  ↓
执行操作（编辑/删除）
  ↓
后端权限校验
  ↓
返回操作结果
```

### 2.4 用户管理（普通用户）

**功能描述**：
普通用户只能管理和修改自己的账户信息，无法访问其他用户的数据。

**权限范围**：
- 查看自己的用户信息
- 修改自己的用户名（可选）
- 修改自己的邮箱（可选）
- 修改自己的头像（可选）
- 无法删除账户
- 无法访问其他用户信息
- 无法修改角色

**操作流程**：
```
普通用户登录系统
  ↓
访问个人信息页面
  ↓
查看自己的用户信息
  ↓
修改个人信息
  ↓
后端权限校验（只能修改自己的信息）
  ↓
返回操作结果
```

### 2.5 数据统计（可选扩展）

**功能描述**：
管理员可以查看系统用户统计数据。

**统计维度**：
- 用户总数
- 管理员数量
- 普通用户数量
- 用户状态分布（启用/禁用）

## 3. 非功能需求

### 3.1 安全性需求

**密码安全**：
- [x] 密码明文存储：不允许
- [x] 密码加密算法：建议使用 bcrypt（当前为明文，待实现）
- [x] 密码复杂度要求：6-13位字母数字组合

**认证授权**：
- [x] 用户身份验证：JWT Token
- [x] 基于角色的访问控制（RBAC）
- [x] API 接口权限校验

**数据安全**：
- [x] SQL 注入防护（使用参数化查询）
- [x] 数据库连接安全
- [x] CORS 跨域控制

### 3.2 性能需求

- 响应时间：API 接口响应时间 < 200ms
- 并发处理：支持至少 100 个并发用户
- 数据库连接池：合理配置连接池大小

### 3.3 可用性需求

- 系统可用性：99.9%
- 错误处理：友好且清晰的错误提示
- 数据一致性：ACID 事务保证

### 3.4 可扩展性需求

- 模块化设计，便于功能扩展
- 分层架构（Handler/Service/Repository），便于维护
- 数据库结构设计合理，支持后续扩展

## 5. 数据库设计

### 5.1 数据表：users

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL UNIQUE COMMENT '用户名',
    password VARCHAR(255) NOT NULL COMMENT '密码（加密存储）',
    email VARCHAR(100) DEFAULT NULL COMMENT '邮箱',
    avatar_url VARCHAR(255) DEFAULT NULL COMMENT '头像URL',
    role VARCHAR(20) NOT NULL DEFAULT 'user' COMMENT '角色（admin/user）',
    status INT DEFAULT 1 COMMENT '状态（1启用，0禁用）',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    INDEX idx_username (username),
    INDEX idx_role (role)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

### 5.2 字段说明

| 字段名 | 类型 | 说明 | 示例 |
|--------|------|------|------|
| id | BIGINT | 主键，自增 | 1, 2, 3 |
| username | VARCHAR(50) | 用户名，唯一 | admin, user123 |
| password | VARCHAR(255) | 密码（加密后） | $2a$10$abcd... |
| email | VARCHAR(100) | 邮箱 | user@example.com |
| avatar_url | VARCHAR(255) | 头像URL | /static/avatar/1.jpg |
| role | VARCHAR(20) | 角色 | admin（管理员）、user（普通用户） |
| status | INT | 状态 | 1（启用）、0（禁用） |
| created_at | TIMESTAMP | 创建时间 | 2026-03-04 10:30:00 |
| updated_at | TIMESTAMP | 更新时间 | 2026-03-04 11:00:00 |

### 5.3 索引设计

| 索引名 | 字段 | 类型 | 说明 |
|--------|------|------|------|
| PRIMARY KEY | id | 主键索引 | 主键索引 |
| INDEX | username | 普通索引 | 加速用户名查询 |
| INDEX | role | 普通索引 | 加速角色筛选 |

## 6. API 接口定义

### 6.1 注册接口

**接口描述**：用户注册新账户

**请求信息**：
- **请求方式**：POST
- **请求URL**：`/users`
- **请求头**：`Content-Type: application/x-www-form-urlencoded`

**请求参数**：
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| username | string | 是 | 用户名，6-13位字母数字 |
| password | string | 是 | 密码，6-13位字母数字 |
| email | string | 否 | 邮箱 |

**响应示例**：
```json
{
  "code": 200,
  "message": "注册成功!用户名 : admin,用户ID: 1"
}
```

**错误码**：
| 错误码 | 说明 |
|--------|------|
| 400 | 参数错误或用户名已存在 |
| 405 | 不支持的请求方法 |

---

## 7. 权限控制设计

### 7.1 角色定义

| 角色名称 | 角色代码 | 权限说明 |
|---------|---------|---------|
| 管理员 | admin | 完整系统权限，包括用户管理 |
| 普通用户 | user | 个人信息管理权限 |

### 7.2 权限矩阵

| 功能模块 | 管理员 | 普通用户 |
|---------|--------|---------|
| 用户列表查看 | ✅ | ❌ |
| 用户信息修改 | ✅ | ❌（仅自己） |
| 用户删除 | ✅ | ❌ |
| 角色修改 | ✅ | ❌ |
| 个人信息修改 | ✅ | ✅ |
| 账户删除 | ❌ | ❌ |

### 7.3 权限控制实现

**前端权限控制**：
- 根据用户角色动态显示/隐藏按钮
- 管理员看到所有操作按钮
- 普通用户只看到修改自己的按钮

**后端权限校验**：
- 每个敏感操作都需要校验用户角色
- 防止绕过前端直接调用API
- 使用中间件统一处理认证和授权

## 8. 接口安全设计

### 8.1 认证机制

**JWT Token 认证**：
1. 用户登录成功后，后端生成 JWT Token
2. 前端将 Token 存储在 localStorage
3. 每次请求在请求头中携带 Token：`Authorization: Bearer {token}`
4. 后端中间件验证 Token 有效性

### 8.2 防护措施

**SQL 注入防护**：
- 使用参数化查询（`sql.DB` 的 `Exec` 和 `QueryRow` 方法）

**XSS 攻击防护**：
- 前端输出时进行 HTML 转义（使用 Go template 的 `html/template`）

**CSRF 攻击防护**：
- 使用 Token 验证（可在后续实现）

**CORS 跨域控制**：
- 配置 CORS 中间件，允许指定域名访问
