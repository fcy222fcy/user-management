# 依赖注入

 Go 语言中，**依赖注入（Dependency Injection, 简称 DI）** 是一种设计模式，其核心思想是：**一个对象（或函数）不应该自己创建它所依赖的其他对象，而是应该由外部将这些依赖“传递”给它。**
这与传统的“在内部 `new` 出来”的方式相反。Go 语言由于没有类（Class）和复杂的反射框架（如 Java 的 Spring），其依赖注入通常非常**显式、简单且直接**，主要依靠**构造函数传参**或**结构体字段赋值**来实现。

## 对比

### 没有使用依赖注入(硬编码依赖)
在这种方式下,组件内部直接创建了他需要的依赖,这导致代码耦合度高,难以测试

```Go
type MySQLDB struct{} 

func (m *MySQLDB) Query(sql string) string { 
	return "data from mysql" 
} 

type UserService struct { 
	// 内部直接硬编码了依赖 
	db *MySQLDB 
} 

// 问题：NewUserService 内部自己 new 了 DB，无法轻易替换为 Mock DB 
func NewUserService() *UserService { 
	return &UserService{ 
		db: &MySQLDB{}, // 硬编码依赖 
	} 
} 

func (s *UserService) GetUser(id int) string { 
	return s.db.Query("SELECT * FROM users WHERE id = ?") 
}
```

### 使用依赖注入(外部传入依赖)
在这种方式下，`UserService` 只声明它需要一个符合特定接口（Interface）的对象，具体是哪个实现由外部决定。
```Go
// 1. 定义接口 (Go 的核心：面向接口编程) 
type Database interface { 
	Query(sql string) string 
} 

type MySQLDB struct{}
 
func (m *MySQLDB) Query(sql string) string { 
	return "data from mysql" 
} 
// MockDB 用于测试 
type MockDB struct{} 

func (m *MockDB) Query(sql string) string { 
	return "mock data" 
} 

type UserService struct { 
	db Database // 依赖的是接口，而不是具体实现 
} 

// 2. 构造函数注入 (Constructor Injection) - Go 中最常用的方式 
// 外部调用者负责传入具体的 db 实现 
func NewUserService(db Database) *UserService { 
	return &UserService{ db: db, } 
} 

func (s *UserService) GetUser(id int) string { 
	return s.db.Query("SELECT * FROM users WHERE id = ?") 
}
```

## Go中依赖注入的常见形式
### 构造函数注入(Constructor Injection)
上面例子就是, 通过`New...`函数的参数传递依赖

> 优点: 依赖关系在代码签名中一目了然,编译期检查,不可变(通常配合只读字段)
### 结构体字段注入(Setter/Field Injection)
先创建结构体,再手动赋值字段
```Go
svc := &UserService{}
svc.db = &MySQLDB{} //外部手动赋值
```

> 缺点: 对象可能处于"未初始化完成" 的中间状态,不如构造函数安全
### 使用第三方 DI 容器
对于超大型项目，手动组装依赖树可能很繁琐。这时可以使用如 **Google Wire**, **Uber FX**, **Dig** 等工具。

- **Wire**: 是一个代码生成工具。它在编译时生成组装代码，如果没有找到依赖会直接编译报错，没有运行时反射开销。
- **FX/Dig**: 基于反射，在运行时解析依赖图。

## 依赖注入的好处
1. 解耦:
	`UserService` 不再关心 `db` 具体是 MySQL、PostgreSQL 还是 Redis。只要实现了 `Database` 接口即可。更换数据库实现时，无需修改 `UserService` 的代码。
2. 易于测试:
	在单元测试中,你可以轻松传入一个`MockDB`,而不需要真正连接数据库
```Go
	func TestGetUser(t *testing.T) { 
		// 注入 Mock 依赖 
			mockDB := &MockDB{} 
		svc := NewUserService(mockDB) 
	
		result := svc.GetUser(1) 
		if result != "mock data" { 
			t.Fail() 
		} 
	}
```

# 业务
在完成项目的时候,一直再说 service是处理业务,到底什么是业务呢

问了下claude code,回答如下

在软件开发中,"业务"指

1. 业务逻辑(核心功能)

- 比如: 用户注册需要校验邮箱格式,密码强度
- 比如:管理员可以删除用户,但普通用户不能
- 这些都是特定于你应用的功能,不是通用的技术功能
- 还有就是,DAO层有删除用户的功能,这里就只管删除用户,不再判断是不是管理员,因为这需要再 Service层判断

2. 业务规则

- 比如: 新用户注册送100 积分
- 比如: 积分只能使用一次,不能重复使用(业务规则)

3. 业务流程

- 比如:用户从登录到下单的完成流程


## 总之

"业务"就是你的应用具体要解决什么问题,实现什么功能:

比如这个项目是"用户管理系统",那么:
- 用户注册/登录是业务
- 用户信息管理是业务
- 权限控制是业务
- 数据验证是业务

技术实现只是工具,业务是目的



# CORS 跨域

要理解这个,首先要理解"同源策略"

浏览器有个安全规则: 不同域名/端口/协议的页面,不能互相访问对方的数据

> 同源 = 域名 + 端口 + 协议 都相同
> 跨域 = 只要有一个不同



| URL                   | 同源?                           |
| --------------------- | ----------------------------- |
| `https://a.com/page1` | 同源                            |
| `https://b.com/page1` | x 跨域(域名不同)                    |
| `https://a.com:8080`  | x 跨域(端口不同)                    |
| `http://a.com/page1`  | x 跨域(协议不同,一个是 https ,一个是http) |

举个例子：
场景1：没有CORS（前端在后端同域名）

```Go
// 前端：https://a.com
// 后端：https://a.com/api/login

  fetch('https://a.com/api/login', {  // 同源，可以访问
      method: 'POST',
      body: JSON.stringify({username: 'test'})
  })
```

  ✅ 成功 - 同源，没有跨域问题
  
  ---
  场景2：有CORS（前后端分离）
  
```Go
  // 前端：https://a.com
  // 后端：https://api.b.com

  fetch('https://api.b.com/api/login', {  // 跨域      
      method: 'POST',
      body: JSON.stringify({username: 'test'})
  })
```

  ❌ 浏览器拦截 - 因为是跨域请求

---
CORS的作用 ?????

CORS不是浏览器阻止请求,而是后端告诉浏览器: "我可以让你访问"


为什么需要CORS？

  不加CORS时：
```Go
  // 前端发请求
  fetch('https://api.b.com/login')

  // 浏览器：
  // ❌ "这跨域了，我不发这个请求！"
  // （根本不会到后端）

```

  加了CORS后：
```Go
  // 后端：
  // ✅ "没问题，允许跨域"
  c.Writer.Header().Set("Access-Control-Allow-Origin", 
  "*")

  // 前端：
  // ✅ "好，服务器允许，那我发请求"
  fetch('https://api.b.com/login')  // 请求成功    
```

# Token

可以用于 登录鉴权/路由守卫,即 用户如果没有登录,只能访问登录页,登陆成功之后,才能访问其他页面
整体流程
1️⃣ 用户访问网站  
2️⃣ 前端检查 **有没有登录 token**  
3️⃣ 如果没有 token → **跳转登录页**  
4️⃣ 登录成功 → 后端返回 **token**  
5️⃣ 前端把 token 存起来（localStorage / cookie）  
6️⃣ 再访问其他页面时，检测 token  
7️⃣ 有 token → 允许进入页面

同时也需要后端的 JWT 中间件来配合
```text
用户访问 /home
      │
      ▼
前端检查 token
      │
      ├ 没有 → 跳转 /login
      │
      └ 有 → 访问接口

前端请求接口
      │
      ▼
Go 后端 JWT 中间件
      │
      ├ token无效 → 401
      │
      └ token有效 → 返回数据
```

# 中间件
我一直不了解这个中间件的作用,然后今天从 https://chai2010.cn/advanced-go-programming-book/ch5-web/ch5-03-middleware.html 这一篇博客上看到了这个介绍,也是清楚一些了

## 代码泥潭
先来看一段代码示例
```Go
// middleware/hello.go
package main

func hello(wr http.ResponseWriter, r *http.Request) {
    wr.Write([]byte("hello"))
}

func main() {
    http.HandleFunc("/", hello)
    err := http.ListenAndServe(":8080", nil)
    ...
}

```
这是一个典型的 Web 服务，挂载了一个简单的路由。我们的线上服务一般也是从这样简单的服务开始逐渐拓展开去的。

现在突然来了一个新的需求，我们想要统计之前写的 hello 服务的<span style="color:red">处理耗时</span>，需求很简单，我们对上面的程序进行少量修改：
```Go
// middleware/hello_with_time_elapse.go
var logger = log.New(os.Stdout, "", 0)

func hello(wr http.ResponseWriter, r *http.Request) {
    timeStart := time.Now()
    wr.Write([]byte("hello"))
    timeElapsed := time.Since(timeStart)
    logger.Println(timeElapsed)
}

```
这样便可以在每次接收到 http 请求时，打印出当前请求所消耗的时间。

完成了这个需求之后，我们继续进行业务开发，提供的 API 逐渐增加，现在我们的路由看起来是这个样子：
```Go
// middleware/hello_with_more_routes.go
// 省略了一些相同的代码
package main

func helloHandler(wr http.ResponseWriter, r *http.Request) {
    // ...
}

func showInfoHandler(wr http.ResponseWriter, r *http.Request) {
    // ...
}

func showEmailHandler(wr http.ResponseWriter, r *http.Request) {
    // ...
}

func showFriendsHandler(wr http.ResponseWriter, r *http.Request) {
    timeStart := time.Now()
    wr.Write([]byte("your friends is tom and alex"))
    timeElapsed := time.Since(timeStart)
    logger.Println(timeElapsed)
}

func main() {
    http.HandleFunc("/", helloHandler)
    http.HandleFunc("/info/show", showInfoHandler)
    http.HandleFunc("/email/show", showEmailHandler)
    http.HandleFunc("/friends/show", showFriendsHandler)
    // ...
}

```

渐渐的我们的系统增加到了 30 个路由和 handler 函数，每次增加新的 handler，我们的第一件工作就是把之前写的所有和业务逻辑无关的周边代码先拷贝过来。

接下来系统安稳地运行了一段时间，突然有一天，老板找到你，我们最近找人新开发了监控系统，为了系统运行可以更加可控，需要把每个接口运行的耗时数据主动上报到我们的监控系统里。给监控系统起个名字吧，叫 metrics。现在你需要修改代码并把耗时通过 HTTP Post 的方式发给 metrics 系统了。我们来修改一下 `helloHandler()`：
```Go
func helloHandler(wr http.ResponseWriter, r *http.Request) {
    timeStart := time.Now()
    wr.Write([]byte("hello"))
    timeElapsed := time.Since(timeStart)
    logger.Println(timeElapsed)
    // 新增耗时上报
    metrics.Upload("timeHandler", timeElapsed)
}

```
修改到这里,本能地发现我们的开发工作开始陷入了泥潭。无论未来对我们的这个 Web 系统有任何其它的非功能或统计需求，我们的修改必然牵一发而动全身。

## 使用中间件剥离非业务逻辑(解耦)

我们来分析一下，一开始在哪里做错了呢？我们只是一步一步地满足需求，把我们需要的逻辑按照流程写下去呀？

我们犯的最大的错误，是把业务代码和非业务代码揉在了一起。对于大多数的场景来讲，非业务的需求都是在 http 请求处理前做一些事情，并且在响应完成之后做一些事情。
这个时候就想到了解耦

我们需要给我们的 `helloHandler()` 增加超时时间统计，我们可以使用一种叫 `function adapter` 的方法来对 `helloHandler()` 进行包装：
```Go

func hello(wr http.ResponseWriter, r *http.Request) {
    wr.Write([]byte("hello"))
}

func timeMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(wr http.ResponseWriter, r *http.Request) {
        timeStart := time.Now()

        // next handler
        next.ServeHTTP(wr, r)

        timeElapsed := time.Since(timeStart)
        logger.Println(timeElapsed)
    })
}

func main() {
    http.Handle("/", timeMiddleware(http.HandlerFunc(hello)))
    err := http.ListenAndServe(":8080", nil)
    ...
}

```


这样就非常轻松地实现了业务与非业务之间的剥离，魔法就在于这个 `timeMiddleware`。可以从代码中看到，我们的 `timeMiddleware()` 也是一个函数，其参数为 `http.Handler`，`http.Handler` 的定义在 `net/http` 包中：
```Go
type Handler interface {
    ServeHTTP(ResponseWriter, *Request)
}

```

任何方法实现了 `ServeHTTP`，即是一个合法的 `http.Handler`，读到这里你可能会有一些混乱，我们先来梳理一下 http 库的 `Handler`，`HandlerFunc` 和 `ServeHTTP` 的关系：
```Go
type Handler interface {
    ServeHTTP(ResponseWriter, *Request)
}

type HandlerFunc func(ResponseWriter, *Request)

func (f HandlerFunc) ServeHTTP(w ResponseWriter, r *Request) {
    f(w, r)
}


```
所以,只要你的 handler 函数签名是:
```Go
func (ResponseWriter, *Request)

```
那么这个 handler 和 `http.HandlerFunc()` 就有了一致的函数签名，可以将该 `handler()` 函数进行类型转换，转为 `http.HandlerFunc`。而 `http.HandlerFunc` 实现了 `http.Handler` 这个接口。在 `http` 库需要调用你的 handler 函数来处理 http 请求时，会调用 `HandlerFunc()` 的 `ServeHTTP()` 函数，可见一个请求的基本调用链是这样的：
```go
h = getHandler() => h.ServeHTTP(w, r) => h(w, r)
```
上面提到的把自定义 `handler` 转换为 `http.HandlerFunc()` 这个过程是必须的，因为我们的 `handler` 没有直接实现 `ServeHTTP` 这个接口。上面的代码中我们看到的 HandleFunc(注意 HandlerFunc 和 HandleFunc 的区别) 里也可以看到这个强制转换过程：
```go
func HandleFunc(pattern string, handler func(ResponseWriter, *Request)) {
    DefaultServeMux.HandleFunc(pattern, handler)
}

// 调用

func (mux *ServeMux) HandleFunc(pattern string, handler func(ResponseWriter, *Request)) {
    mux.Handle(pattern, HandlerFunc(handler))
}

```
知道 handler 是怎么一回事，我们的中间件通过包装 handler，再返回一个新的 handler 就好理解了。

总结一下，我们的中间件要做的事情就是通过一个或多个函数对 handler 进行包装，返回一个包括了各个中间件逻辑的函数链。我们把上面的包装再做得复杂一些：
```go
customizedHandler = logger(timeout(ratelimit(helloHandler)))
```
这个函数链在执行过程中的上下文如下图所示
![[Pasted image 20260306170837.png]]
再直白一些，这个流程在进行请求处理的时候就是不断地进行函数压栈再出栈，有一些类似于递归的执行流：

```text
[exec of logger logic]		   函数栈: []

[exec of timeout logic]		  函数栈: [logger]

[exec of ratelimit logic]		函数栈: [timeout/logger]

[exec of helloHandler logic]	 函数栈: [ratelimit/timeout/logger]

[exec of ratelimit logic part2]  函数栈: [timeout/logger]

[exec of timeout logic part2]	函数栈: [logger]

```
功能实现了，但在上面的使用过程中我们也看到了，这种函数套函数的用法不是很美观，同时也不具备什么可读性。

## 更优雅的中间件写法

上一节中解决了业务功能代码和非业务功能代码的解耦，但也提到了，看起来并不美观，如果需要修改这些函数的顺序，或者增删中间件还是有点费劲，本节我们来进行一些 “写法” 上的优化。
看一个例子:
```Go
r = NewRouter()
r.Use(logger)
r.Use(timeout)
r.Use(ratelimit)
r.Add("/", helloHandler)

```
通过多步设置，我们拥有了和上一节差不多的执行函数链。胜在直观易懂，如果我们要增加或者删除中间件，只要简单地增加删除对应的 `Use()` 调用就可以了。非常方便。

从框架的角度来讲，怎么实现这样的功能呢？也不复杂：
```go
type middleware func(http.Handler) http.Handler

type Router struct {
    middlewareChain [] middleware
    mux map[string] http.Handler
}

func NewRouter() *Router {
    return &Router{
        mux: make(map[string]http.Handler),
    }
}

func (r *Router) Use(m middleware) {
    r.middlewareChain = append(r.middlewareChain, m)
}

func (r *Router) Add(route string, h http.Handler) {
    var mergedHandler = h

    for i := len(r.middlewareChain) - 1; i >= 0; i-- {
        mergedHandler = r.middlewareChain[i](mergedHandler)
    }

    r.mux[route] = mergedHandler
}

```
注意代码中的 `middleware` 数组遍历顺序，和用户希望的调用顺序应该是 "相反" 的。应该不难理解。

```text
  => 设置一个特殊的路由，例如 / ping，/healthcheck，用来给负载均衡一类的前置服务进行探活
logger.go
  => 打印请求处理处理日志，例如请求处理时间，请求路由
profiler.go
  => 挂载 pprof 需要的路由，如 `/pprof`、`/pprof/trace` 到系统中
realip.go
  => 从请求头中读取 X-Forwarded-For 和 X-Real-IP，将 http.Request 中的 RemoteAddr 修改为得到的 RealIP
requestid.go
  => 为本次请求生成单独的 requestid，可一路透传，用来生成分布式调用链路，也可用于在日志中串连单次请求的所有逻辑
timeout.go
  => 用 context.Timeout 设置超时时间，并将其通过 http.Request 一路透传下去
throttler.go
  => 通过定长大小的 channel 存储 token，并通过这些 token 对接口进行限流

```

每一个 Web 框架都会有对应的中间件组件，如果你有兴趣，也可以向这些项目贡献有用的中间件，只要合理一般项目的维护人也愿意合并你的 Pull Request。

# HTTP 和 HTTP关于加密问题

| 特性    | HTTP (超文本传输协议)              | HTTPS (安全超文本传输协议)                            |
| :---- | :-------------------------- | :------------------------------------------- |
| 全称    | HyperText Transfer Protocol | HyperText Transfer Protocol Secure           |
| 安全性   | ❌ 低。数据明文传输，容易被窃听、篡改、冒充。     | ✅ 高。数据加密传输，防窃听、防篡改、防冒充。                      |
| 端口    | 默认 80                       | 默认 443                                       |
| 协议层   | 直接运行在 TCP 之上                | 在 TCP 和 HTTP 之间加了 SSL/TLS 层                  |
| 证书    | 不需要                         | 需要向 CA 机构申请 数字证书 (通常收费，也有免费的如 Let's Encrypt) |
| 性能    | 稍快 (无加密开销)                  | 略慢 (需握手和加解密，但现代硬件下差异可忽略，且 HTTP/2 必须基于 HTTPS) |
| SEO   | 普通                          | 搜索引擎 (Google/百度) 优先收录                        |
| 浏览器标识 | 显示“不安全”或无特殊标识               | 显示 🔒 小锁图标                                   |

###  HTTPS 是如何加密的？（混合加密机制）

HTTPS 并不是全程只用一种加密方式，而是聪明地结合了 **非对称加密** 和 **对称加密** 的优点。

#### 为什么需要两种加密？

- **非对称加密 (公钥/私钥)**：安全性极高，但**速度慢**，计算量大。适合用来交换密钥。
- **对称加密 (同一个密钥)**：**速度极快**，适合大量数据传输，但前提是双方得先安全地把密钥商量好。

####  TLS 握手流程 (简化版)

当你访问 `https://www.example.com` 时，发生了以下过程：

1. **客户端发起请求 (Client Hello)**：
    - 浏览器告诉服务器：“我想建立安全连接，这是我支持的加密算法列表。”
2. **服务器回应 (Server Hello + 证书)**：
    - 服务器说：“好的，我们用这套算法。这是我的**身份证 (数字证书)**，里面包含我的**公钥**。”
    - _注：证书由权威的 CA 机构签发，浏览器会验证证书是否合法，防止假冒网站。_
3. **验证与生成密钥**：
    - 浏览器验证证书合法后，生成一个随机的 **对称密钥 (Session Key)**。
    - 浏览器用服务器的 **公钥** 把这个 **对称密钥** 加密，发送给服务器。
    - _此时，即使黑客截获了数据，因为没有服务器的私钥，也打不开这个包，拿不到对称密钥。_
4. **服务器解密**：
    - 服务器用自己的 **私钥** 解密，拿到了 **对称密钥**。
5. **开始加密通信**：
    - **从此之后**，双方都使用这个 **对称密钥** 进行快速加密和解密所有数据（包括你的密码、Cookie、聊天内容等）。

> **形象比喻**：
> 
> - **非对称加密** 就像你有一个公开的信箱（公钥），任何人都能往里面投信，但只有你有钥匙（私钥）能打开取信。你用这个信箱把“以后我们沟通的暗号”（对称密钥）安全地传给对方。
> - **对称加密** 就像拿到暗号后，你们俩开始用这个暗号写日记，既快又安全。

---

###  HTTP 的三大风险 (为什么要升级 HTTPS)

如果你还在用 HTTP 开发用户管理系统，会面临以下严重问题：

1. **窃听 (Eavesdropping)**：
    - **场景**：用户在公共 WiFi 下登录你的系统。
    - **后果**：黑客使用抓包工具 (如 Wireshark, Fiddler) 可以**直接看到明文**的用户名和密码。你在刚才的 Network 面板里看到的明文数据，在 HTTP 下，任何经过的路由器都能看到。
2. **篡改 (Tampering)**：
    - **场景**：运营商或黑客在传输路径上修改数据。
    - **后果**：你网页里插入的广告、甚至把下载按钮改成病毒链接。对于 API 返回的 JSON 数据，黑客可以修改 `{"role": "user"}` 为 `{"role": "admin"}`。
3. **冒充 (Impersonation)**：
    - **场景**：黑客搭建一个和你一模一样的钓鱼网站。
    - **后果**：用户无法分辨真假，因为 HTTP 没有身份认证机制。HTTPS 通过 CA 证书证明了“我真的是我”。



# 前端发送 POST 请求时的 常见的三种数据格式:

今天在测试用户管理系统的,明明数据是正确的,但是在后端反应 用户名不合法,经过一层层排查之后发现,问题出来 r.Parseform()这里,这里解析完之后,值全是空值
,通过这个`fmt.Println("-Content-Type:", r.Header.Get("Content-Type"))`得出结果为`Content-Type: multipart/form-data; boundary=----WebKitFormBoundaryTBuMyH5mdfcsSvGA`,那么真相也就水落石出了--类型不匹配,但是这个 ParseForm()明明解析到了错误的内容,居然不报错;查了ai之后,发现了现在前端发送 POST请求的时候,主要有三种数据格式:

| 方式                                      | Content-Type                        | 适用场景                  | 前端写法                                                                                                                                    | 方式送出来的数据                                                                                                                                                                                                                                                                                                                                                                         | 后端解析方式                                                                                                             | 优缺点                                                  |
| --------------------------------------- | ----------------------------------- | --------------------- | --------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------- |
| **1. JSON**                             | `application/json`                  | 现代前后端分离、发送复杂结构（对象/数组） | `fetch('/login', { method:'POST', headers:{'Content-Type':'application/json'}, body:JSON.stringify(data) })`                            | {<br>  "username":"tom",<br>  "password":"123456"<br>}                                                                                                                                                                                                                                                                                                                           | `json.NewDecoder(r.Body).Decode(&struct{})`                                                                        | ✅ 清晰、易扩展、适合 API❌ 不能直接用 `FormValue()`，文件需另传           |
| **2. URL-encoded**(传统表单)                | `application/x-www-form-urlencoded` | 表单提交、简单文本字段           | `fetch('/login', { method:'POST', headers:{'Content-Type':'application/x-www-form-urlencoded'}, body: new URLSearchParams(formData) })` | username=tom&password=123456<br>                                                                                                                                                                                                                                                                                                                                                 | `r.ParseForm(); r.FormValue("username")`                                                                           | ✅ 后端兼容性好、直接用 `FormValue()`❌ 不能上传文件，数据复杂性受限           |
| **3. Multipart/form-data**(专为文件上传设计的格式) | `multipart/form-data; boundary=xxx` | 上传文件或混合数据             | `fetch('/login', { method:'POST', body: new FormData(form) })`                                                                          | 请求体如下:<br>------WebKitFormBoundaryxxx<br>Content-Disposition: form-data; name="username"<br><br>tom<br>------WebKitFormBoundaryxxx<br>Content-Disposition: form-data; name="password"<br><br>123456<br>------WebKitFormBoundaryxxx<br>Content-Disposition: form-data; name="avatar"; filename="a.png"<br>Content-Type: image/png<br><br>(文件二进制)<br>------WebKitFormBoundaryxxx-- | `r.ParseMultipartForm(10 << 20); r.FormValue("username")`<br><br>如果读取文件<br>file,header,err := r.FormFIle("avatar") | ✅ 支持文件和表单混合❌ 后端解析略复杂，需要 `ParseMultipartForm`，不适合简单文本 |

## 总结:
- **只传账号密码这种简单表单** → 推荐 **URL-encoded**，简单，后端不改就能用
    
- **前后端分离、JSON API** → 用 **JSON**，现代做法，数据结构清晰
    
- **需要上传文件** → 用 **multipart/form-data**

form 可以用 enctype类型

# http.Handler和http.Handlerfunc的区别

## http.Handler 是一个接口
```go
type Handler interface {
    ServeHTTP(ResponseWriter, *Request)
}
```
意思是:
只要有一个类型实现了这个方法,他就可以
```Go
ServeHTTP(ResponseWriter, *Request)
```
他就可以当做 handler使用
例如:
```go
type MyHandler struct{}

func (h MyHandler) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    w.Write([]byte("hello"))
}
```
使用:
```go
http.ListenAndServe(":8080", MyHandler{})
```

所以:
Handler 相当于一种规范(接口)

## http.Handlerfunc 是一个函数类型
```go
type HandlerFunc func(ResponseWriter, *Request)
```
然后它实现了 ServeHTTP方法:
```go
func (f HandlerFunc) ServeHTTP(w ResponseWriter, r *Request) {
    f(w, r)
}
```
就是把普通函数包装成 Handler

## 为什么要有 HandlerFunc
因为写Handler 结构体太麻烦

不用HandlerFunc的方式:
```go
type LoginHandler struct{}

func (h LoginHandler) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    fmt.Println("login")
}
```

用了 HandlerFunc的方式:
```go
func login(w http.ResponseWriter, r *http.Request) {
    fmt.Println("login")
}

http.HandlerFunc("/login",login)
```

# HTTP协议是无状态的 和 Connection:keep-alive 的区别

无状态是指协议对于事务处理没有记忆能力，服务器不知道客户端是什么状态。从另一方面讲，打开一个服务器上的网页和你之前打开这个服务器上的网页之间没有任何联系。

HTTP 是一个无状态的面向连接的协议，无状态不代表 HTTP 不能保持 TCP 连接，更不能代表 HTTP 使用的是 UDP 协议（面对无连接）。

从 HTTP/1.1 起，默认都开启了 Keep-Alive 保持连接特性，简单地说，当一个网页打开完成后，客户端和服务器之间用于传输 HTTP 数据的 TCP 连接不会关闭，如果客户端再次访问这个服务器上的网页，会继续使用这一条已经建立的 TCP 连接。

Keep-Alive 不会永久保持连接，它有一个保持时间，可以在不同服务器软件（如 Apache）中设置这个时间。
